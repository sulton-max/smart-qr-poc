# v0.4 — SDK adoption (backend-beta + frontend-beta)

*Last updated: 2026-06-19*

**Status:** 🚧 In Progress · **Started:** 2026-06-19 · **Completed:** —

> The adoption pass: run smart-qr on the wow-two SDK for everything that isn't business logic, and adopt `@wow-two-beta/ui` fully. **Adopt-first** — consume what the SDK already ships; anything the SDK lacks (new symbols/modules) is deferred to a later extraction version, not built here.

## Decisions (resolved)

- **D1 — error model:** keep `AppResult` + `FailureCategory` (no `DomainError`). The errors / validation / mapping / codes redesign is a **separate future chat**, out of v0.4.
- **D2 — snake_case everywhere:** Postgres-native — table names, columns, **enum stored values** (adopt the SDK snake_case converter → a `QrCode`→`qr_code` data migration), and JSON serialization. *(JSON → snake_case is a wire-contract change; frontend types update in Iter 6 — flag if DB-only was meant.)*
- **D3 — testing package:** publish `WoW2.Sdk.Backend.Beta.Testing` as a **separate** package (keeps xUnit / Testcontainers / Respawn out of the runtime SDK).
- **D4 — frontend:** no `frontend-common` extract — **scan the frontend, replace with `@wow-two-beta/ui` in place**; app-local infra (api client, `usePageMeta`, identity gate) stays app-local.
- **D5 — adopt before extract:** consume the existing SDK surface first (incl. the **entity contracts** — smart-qr still uses its own `IEntity`). The EXTRACT halves that need new SDK code (Google ID-token verifier, guest-session, the test-harness gaps, the `ApiResponse<T>` Web module, the startup-migrate adapter) are **deferred** to a later version.

### Iteration 1 — Host floor: boot on `AddApiDefaults` ✅

- [x] Ability to wire both hosts via `AddApiDefaults`/`UseApiDefaults` (Serilog · OTel · health · ProblemDetails · OpenAPI), replacing the hand-rolled per-area `Add*`/`Use*`.
- [x] Ability to drop the custom `/health` and startup banners; output-cache + rate-limiting off on both hosts (hot-path-safe). *(CORS + JSON string-enums kept product-side — the SDK CORS can't express the cookie-auth credentials and `AddApiDefaults` registers no controllers; both revisit when the SDK adds them — deferred per D5.)*

### Iteration 2 — Persistence base + entity contracts + snake_case

- [ ] Ability to run on `AppDbContextBase` + the SDK audit interceptor, dropping the hand-rolled timestamps and conventions.
- [ ] Ability to use the SDK entity contracts (`IKeyedEntity<Guid>` / `IAuditable` / `IHasTableName`) instead of the local `IEntity`.
- [ ] Ability to persist enums as snake_case (the data migration + SQL/seed update).

### Iteration 3 — Presentation + validation from the SDK

- [ ] Ability to return trace-correlated ProblemDetails via the SDK handler.
- [ ] Ability to validate requests through the SDK FluentValidation + mediator behavior (author the create / update / checkout / sign-in validators).
- [ ] Ability to serialize the API in snake_case (the SDK JSON preset).

### Iteration 4 — Identity: adopt the SDK cookie scheme

- [ ] Ability to run cookie auth via the SDK `AddCookieAuthentication` with a thin local 401/403 override for the SPA.
- (Google ID-token verifier + guest-session extraction → deferred per D5; the seams stay inline.)

### Iteration 5 — Test baseline: adopt the published SDK Testing package

- [ ] Ability to run the suites on the published `WoW2.Sdk.Backend.Beta.Testing` package, retiring `SmartQr.Platform.Testing` (a verbatim fork).
- (Respawn reset / poll-until / PG-migrator-base extraction → deferred; they stay in the product fixture.)

### Iteration 6 — Frontend: adopt `@wow-two-beta/ui` in place

- [ ] Ability to render shells, accordion, sections, stats, and alerts from the beta UI, deleting the hand-rolled equivalents; update the contract types to snake_case.

### Iteration 7 — Verification: green on the adopted stack

- [ ] Build both hosts + run unit + E2E (Testcontainers) → all green.
- [ ] Smoke the live app (guest → Google → claim → cross-device) — behaviour unchanged.
