# v0.4 — SDK adoption (backend-beta + frontend-beta)

*Last updated: 2026-06-19*

**Status:** 🚧 In Progress · **Started:** 2026-06-19 · **Completed:** —

> The adoption pass: run smart-qr on the wow-two SDK for everything that isn't business logic, and adopt `@wow-two-beta/ui` fully. **Adopt-first** — consume what the SDK already ships; anything the SDK lacks (new symbols/modules) is deferred to a later extraction version, not built here.

## Decisions (resolved)

- **D1 — error model:** keep `AppResult` + `FailureCategory` (no `DomainError`). The errors / validation / mapping / codes redesign is a **separate future chat**, out of v0.4.
- **D2 — casing:** **snake_case in the DB only** (tables · columns · enum stored values — done in iter 2 via the SDK converter + `004` migration). The **API↔frontend JSON wire stays camelCase** (properties + C#-name string-enums, unchanged). snake_case is strictly a DB↔API concern.
- **D3 — testing package:** publish `WoW2.Sdk.Backend.Beta.Testing` as a **separate** package (keeps xUnit / Testcontainers / Respawn out of the runtime SDK).
- **D4 — frontend:** no `frontend-common` extract — **scan the frontend, replace with `@wow-two-beta/ui` in place**; app-local infra (api client, `usePageMeta`, identity gate) stays app-local.
- **D5 — close gaps in the SDK in-version (reversed):** consume what the published SDK already ships immediately; when the SDK **lacks** something (CORS credentials · Google ID-token verifier · guest-session · test-harness helpers · the `ApiResponse<T>` Web module · the startup-migrate adapter), **add it to the SDK source** — the owner reviews / pushes / publishes a new version, then the app bumps + adopts it. No deferral; new additions just take a publish round-trip. Still: adopt the **entity contracts** (smart-qr's own `IEntity` → SDK `IKeyedEntity`).
- **Comments:** teaching-grade — generous XML `<summary>`/`<remarks>` + inline rationale (the *why*, design, gotchas, tradeoffs), not terse one-liners.

### Iteration 1 — Host floor: boot on `AddApiDefaults` ✅

- [x] Ability to wire both hosts via `AddApiDefaults`/`UseApiDefaults` (Serilog · OTel · health · ProblemDetails · OpenAPI), replacing the hand-rolled per-area `Add*`/`Use*`.
- [x] Ability to drop the custom `/health` and startup banners; output-cache + rate-limiting off on both hosts (hot-path-safe). *(CORS + JSON string-enums kept product-side for now — the SDK CORS can't express the cookie-auth credentials and `AddApiDefaults` registers no controllers; per D5 these become SDK additions: credentials-capable CORS + a controllers/JSON preset → then adopt.)*

### Iteration 2 — Persistence base + entity contracts + snake_case ✅

- [x] Ability to run on `AppDbContextBase` + the SDK audit interceptor, dropping the hand-rolled timestamps and conventions.
- [x] Ability to use the SDK entity contracts (`IKeyedEntity<Guid>` / `IAuditable` / `IHasTableName`) instead of the local `IEntity`.
- [x] Ability to persist enums as snake_case (`004-enum-snake-case` migration + the SDK `EnumCaseConverter`). *(DB storage only — JSON wire stays camelCase, unchanged.)*

### Iteration 3 — Presentation + validation from the SDK

- [ ] Ability to return trace-correlated ProblemDetails via the SDK handler.
- [ ] Ability to validate requests through the SDK FluentValidation + mediator behavior (author the create / update / checkout / sign-in validators).

### Iteration 4 — Identity: adopt the SDK cookie scheme

- (Google ID-token verifier + guest-session extraction → deferred per D5; the seams stay inline.)

### Iteration 6 — Frontend: adopt `@wow-two-beta/ui` in place
