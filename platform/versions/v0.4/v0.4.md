# v0.4 — SDK adoption (backend-beta + frontend-beta)

*Last updated: 2026-06-21*

**Status:** 🚧 In Progress · **Started:** 2026-06-19 · **Completed:** —

> The adoption pass: run smart-qr on the wow-two SDK for everything that isn't business logic, and adopt `@wow-two-beta/ui` fully. **Adopt-first** — consume what the SDK already ships; anything the SDK lacks (new symbols/modules) is deferred to a later extraction version, not built here.

## Decisions (resolved)

- **D1 — error model:** keep `AppResult` + `FailureCategory` (no `DomainError`). The errors / validation / mapping / codes redesign is a **separate future chat**, out of v0.4.
- **D2 — casing:** **snake_case in the DB only** (tables · columns · enum stored values — done in iter 2 via the SDK converter + `004` migration). The **API↔frontend JSON wire stays camelCase** (properties + C#-name string-enums, unchanged). snake_case is strictly a DB↔API concern.
- **D3 — testing package:** publish `WoW2.Sdk.Backend.Beta.Testing` as a **separate** package (keeps xUnit / Testcontainers / Respawn out of the runtime SDK).
- **D4 — frontend:** no `frontend-common` extract — **scan the frontend, replace with `@wow-two-beta/ui` in place**; app-local infra (api client, `usePageMeta`, identity gate) stays app-local.
- **D5 — close gaps in the SDK in-version (reversed):** consume what the published SDK already ships immediately; when the SDK **lacks** something (CORS credentials · Google ID-token verifier · guest-session · test-harness helpers · the `ApiResponse<T>` Web module · the startup-migrate adapter), **add it to the SDK source** — the owner reviews / pushes / publishes a new version, then the app bumps + adopts it. No deferral; new additions just take a publish round-trip. Still: adopt the **entity contracts** (smart-qr's own `IEntity` → SDK `IKeyedEntity`).
- **Comments:** teaching-grade — generous XML `<summary>`/`<remarks>` + inline rationale (the *why*, design, gotchas, tradeoffs), not terse one-liners.

### Iteration 3 — Presentation + validation from the SDK ✅

- [x] Trace-correlated ProblemDetails — `AddApiDefaults` already wires `AddTraceAwareProblemDetails` + `AddValidationExceptionHandler` (since iter 1). **Gap found:** the SDK's middleware `ValidationExceptionHandler` doesn't render for an MVC endpoint — during `UseExceptionHandler` the endpoint is cleared, so both problem-details writers decline, `TryWriteAsync` returns false, and the exception re-throws (a 500 in prod / surfaced exception in TestServer). Bridged app-side with an MVC `ValidationExceptionFilter` (renders the 400 inside the action invoker via `ProblemDetailsFactory`, so trace enrichment is preserved). → **shipped to the SDK and adopted**: published in `10.0.28-beta` (`Web.ExceptionHandling` + `AddValidationExceptionFilter()` on `IMvcBuilder`); app swapped to `AddControllers().AddValidationExceptionFilter()` and deleted the local copy.
- [x] Requests validated through the SDK FluentValidation + mediator behavior (`AddMediatorValidationBehavior` + `o.ValidatorAssemblies`). Input-shape validators authored: `CodeCreate` / `CodeUpdate` (name length, fallback + rule destination must be absolute http(s), rule order ≥ 0; shared `RuleDtoValidator`), `BillingCheckout` (plan in enum — Free-eligibility stays in the handler), `GoogleSignIn` (token present). Two E2E negatives added (blank name · non-http url → 400).
- [x] **`ApiResponse<T>` adopted** (D5): the SDK already ships `WoW.Two.Sdk.Backend.Beta.Web.Contracts.ApiResponse<T>` (identical to the local copy) — deleted `SmartQr.Platform.Core/Models/ApiResponse.cs`, swapped the using in the 4 controllers.
- [x] **Layer-DI cleanup** (pending decision → *Codes + Migrate now, CORS in iter 4*): inlined `AddSmartQrCodes` and `MigrateDatabaseAsync` into the host(s), deleted both layer extensions. `AddCorsPolicy` left for iter 4 (reworked there into SDK credentialed-CORS).
- **SDK bump 10.0.24 → 10.0.28-beta** (10.0.26 for the `ApiResponse<T>` + validation symbols; 10.0.28 to adopt the SDK validation filter). Two breaking changes handled: SDK added `Mediator.Result.FailureCategory` (collides with the app's own) → `global using` alias per **D1** in `SmartQr.Api` + `SmartQr.Tests`; `IMigrationDialect` gained `QualifyHistoryTable` → implemented in the `NoOpMigrationDialect` test fake.
- **Cleanup pass** — fixated workspace conventions `code-style/naming.md` (no brand/product prefix on type/member/extension names; predicates `Is`/`Has`/`Can`, never `Be`) and `documentation.md` (strip rationale/justification notes). ### Iteration 4 — Identity: adopt the SDK cookie scheme ✅

- [x] Cookie auth via the SDK `AddCookieAuthentication(o => o.Mode = AuthChallengeMode.Api)` — `Mode=Api` **is** the 401/403 SPA override (no local override needed; cleaner than planned). `CookieName="sqr-auth"`, 30-day sliding; HttpOnly/Secure/`SameSite=Lax` from SDK defaults. Default cookie scheme → `AuthController` `SignInAsync`/`SignOutAsync` + `[Authorize]` unchanged. Replaced the hand-rolled `AddCookie` + `OnRedirectToLogin`/`AccessDenied` events.
- **SDK bump `10.0.28 → 10.0.31-beta`** (5 csprojs) — non-breaking (`29` added the identity layer, `30` fixed cookie options, `31` added gh/container clients; nothing smart-qr consumes changed).
- **CORS — added to SDK + stripped from app.** smart-qr is same-origin (Vite proxies `/api`; prod serves `wwwroot`), so the credentialed policy was dead weight: deleted `CorsExtensions` / `CorsSettings` / `CorsSettingsExtensions` + `AddCustomCors` / `UseCors` + the `appsettings` `Cors` section (grep-clean). Credentialed CORS now lives in the SDK (`Web/Cors.AddCredentialedCorsPolicy(params origins)`, throws on empty → wildcard+credentials impossible) — **uncommitted**, publishes a `10.0.3x-beta` smart-qr can opt into for a split deploy (G2 closed).
- **Single-host build chain** — `BuildSpa` MSBuild target in `SmartQr.Api.csproj` (`BeforeTargets="Build"`, incremental via `Inputs`/`Outputs`) runs `pnpm build` → `wwwroot` on every backend build; no-op when the SPA is unchanged. Proven: touching a SPA input fires `vite` during `dotnet build`.
- **Gaps kept inline per D5:** SPA Google **ID-token verifier** (G1 — SDK ships only server-redirect Google OAuth) and **guest session** (G3 — no SDK anonymous-session primitive) stay product-side. **Claims normalization deferred** (optional; low payoff for a single-provider app). **Default-deny skipped** — smart-qr is guest-first (anon/guests legitimately hit endpoints), so a blanket fallback-deny doesn't fit; per-endpoint `[Authorize]` retained.
- Green: 0 build err · `52` unit · `26` E2E (incl. `AuthTests` cookie scheme) · `10` migrations.
