# v0.6 — Code-rendering SDK extraction

*Last updated: 2026-06-25*

**Status:** 🚧 In progress · **Type:** Adoption · **Started:** 2026-06-25 · **Completed:** —

> Extract the `SmartQr.Codes` render engine + style contract to the backend-beta SDK, adopt it back, then simplify the builder layout — behavior unchanged (byte-identical render, green tests).

## Decisions (locked 2026-06-25)

- **Packaging — mono-lib folder area.** Lands as `src/Codes/` inside `WoW2.Sdk.Backend.Beta` (folder ⇒ namespace `WoW.Two.Sdk.Backend.Beta.Codes`), not a companion package. Cost accepted: `SkiaSharp` native assets + Docker `libfontconfig1` enter the core mono-lib's transitive closure — consistent with the mono-lib's stated "large closure, split when mature" stance. A `.Skia` split is deferred.
- **Format enums — SDK render enums, product maps.** SDK Codes owns `ImageFormat` (render-output, not persisted) + a render `BarcodeFormat`. Product keeps its domain `BarcodeFormat` on `CodeEntity` (persisted column) and maps domain → render at the existing `CodeMappingExtensions` boundary. SDK never owns a persisted product column.

## Iteration 1 — Code-rendering engine → SDK (additive; product untouched) ✅

- [x] Add `src/Codes/` to the mono-lib: render enums (`BarcodeFormat`, `ImageFormat`, `EccLevel`), style contract (`StyleSpec` + `StyleSpecNormalizer` + `StyleSpecJson` + sub-records), `ModuleMatrix`, `SvgRenderer`, `QrMatrixGenerator` (QRCoder), `SkiaSvgRasterizer` (Svg.Skia), `BarcodeRenderer` (ZXing), `CodeRenderer`/`QrCodeRenderer` facades + their interfaces. — 30 files; namespace `SmartQr.Codes` → `WoW.Two.Sdk.Backend.Beta.Codes`; `SvgRenderer` (543 L) relocated byte-for-byte.

> **Notes.** SDK owns render enums `BarcodeFormat`/`ImageFormat` in `Codes/Models/` (product maps at adoption).
